# Contract: Storage and Retention

## Bucket

`relatorios-exportados`

Rules:

- Private bucket.
- No public access.
- Objects are accessed only through short-lived signed URLs generated by `relatorios-exportacao`.
- Service role key is allowed only inside server-side runtime.

## Object Path

```text
<tipo>/<yyyy>/<exportacao_id>/<arquivo_nome>
```

Example:

```text
financeiro/2026/6e6d3f0a-.../relatorio-financeiro-2026-07-01-2026-07-31-6e6d3f.pdf
```

## File Names

PDF:

```text
relatorio-<tipo>-<data_inicial>-<data_final>-<short_id>.pdf
```

CSV package:

```text
relatorio-<tipo>-<data_inicial>-<data_final>-<short_id>.zip
```

ZIP contents:

```text
resumo.csv
detalhes.csv
```

## Signed URL Policy

- Signed URL expiration should be short, recommended 10 minutes.
- Expiration of signed URL is different from business validity.
- Business validity is controlled by `exportacoes_relatorios.expira_em`.
- A new signed URL can be created only if `autorizar_download_exportacao_relatorio` authorizes the current user.

## Retention

- `expira_em = gerado_em + interval '12 months'`.
- Expired exports stay visible in history as `Expirado`.
- Expired exports cannot generate signed URLs.
- Physical object cleanup after expiration is optional for the MVP and may be added later by scheduled job.

## Storage RLS/Policy Expectations

- Bucket policies must deny public object reads.
- Browser clients should not list or read objects directly.
- Upload is performed by Edge Function after RPC authorization.
- If RLS policies are used on `storage.objects`, they should be scoped to controlled paths and not expose broad authenticated access.

## Failure Handling

- If upload fails, RPC status becomes `Falhou`.
- If upload succeeds but DB conclusion fails, implementation should attempt cleanup of the uploaded object and mark failure when possible.
- No response may claim success unless both Storage upload and DB completion succeed.
